# 수상한 수학도시 리부트 PRD

> 상태: 리부트 v2
> 결론: 기존 `지도에서 사건 선택 → 패널에서 문제 풀이` 방향은 보류한다.
> 새 포맷: 라이브 수리 스테이지 기반 브라우저 퍼즐.

## 1. 리부트 판단

기존 기획은 교육 철학과 장기 문서 구조는 강했지만, 아이가 처음 30초 안에 붙잡히는 놀이가 약했다. “수학을 스토리 속에 넣는다”는 말만으로는 부족하다. 화면에서 아이가 먼저 느끼는 것이 문제 카드라면, 아무리 세계관이 좋아도 수학앱처럼 보인다.

따라서 제품의 북극성을 다시 잡는다.

```text
아이는 수학 탐정이 아니라,
고장 난 장난감 도시의 규칙 부품을 끼워 넣는 정비사다.

30-60초마다 예측하고,
부품을 놓고,
작동 버튼을 누르고,
장치가 웃기게 실패하거나 즉시 고쳐지는 것을 본다.
```

핵심 제품 문장:

> 수상한 수학도시는 초등학생이 수학 개념을 규칙 부품처럼 끼워 넣어 고장 난 도시 장치를 고치는 무료 브라우저 수학 퍼즐이다.

## 2. 실패 원인

| 실패 | 설명 | 새 기준 |
| --- | --- | --- |
| 핵심 손맛 부족 | 좋은 문장과 설정은 있었지만 첫 플레이가 답 고르기로 보였다 | 첫 2분 안에 `작동 -> 실패 -> 수정 -> 성공` 루프를 반드시 경험 |
| 장면보다 패널이 강함 | 지도, 카드, 도감, 단서장이 같은 위계로 경쟁했다 | 화면 주인공은 항상 고장 난 장치 스테이지 |
| 미스터리가 추상적 | 제로와 큰 비밀은 장기 동기지만 첫 진입 동기로는 느리다 | 첫 동기는 “이걸 끼우면 움직일까?” |
| 학습 설계 과잉 | 앱, 도서, 워크북, 운영 계약이 먼저 커졌다 | 첫 장치 루프 검증 전에는 확장 보류 |
| 건강한 몰입의 대체 루프 약함 | 금지선은 좋았지만 즉시 반복하고 싶은 조작이 부족했다 | 보상보다 장치 반응, 복구 재생, 다음 고장 장치가 반복 동기 |

## 3. 유지와 폐기

유지한다.

- 개인정보 없음.
- GitHub Pages 정적 배포.
- 이 브라우저 안 진행 저장만 사용.
- 한 에피소드 5-10분.
- 새 개념 1개 중심.
- 상태 있는 SVG/HTML 오브젝트.
- 키보드, 터치, 큰 글자, 모션 축소 접근성.
- 오답을 장면 반응으로 보여주는 원칙.
- 도서/워크북 확장 가능성.

폐기하거나 뒤로 미룬다.

- `production lock` 판정.
- 객관식 중심 `storyChoice`, `mathChoice` 진행.
- 단서장과 제로 미스터리를 첫 동기로 삼는 구조.
- 능력 카드를 보상처럼 보여주는 구조.
- 시계탑 한 편을 완성 기준본으로 고정한 상태.
- 크로스미디어 완성도를 첫 구현 착수 조건으로 두는 방식.

## 4. 제품 형태

제품 형태는 `인터랙티브 그림책 퍼즐`이다. 자유 이동 RPG도 아니고, 문제 카드 웹앱도 아니다.

```text
도시 지도는 허브
라이브 수리 스테이지는 핵심 플레이
단서장과 도감은 보조 기억 장치
스토리북/워크북은 같은 원천의 후속 표현
```

라이브 수리 스테이지의 기본 구조:

```text
고장 난 장치 보기
→ 숫자/도형/시간 부품 집기
→ 장치 슬롯에 끼우기
→ 작동 버튼 누르기
→ 장치가 맞게 움직이거나 이유 있게 실패
→ 다시 고치기
→ 복구 장면 보기
→ 다음 고장 장치 예고
```

정답 제출 버튼은 쓰지 않는다. 핵심 버튼은 `작동`이다.

## 5. 첫 10분 플레이

### 0:00-0:20

바로 장난감 도시 지도. 시계버스 장치가 덜컹거리며 멈춰 있다. 긴 설명은 없다. 큰 행동은 `고치기`.

### 0:20-1:30

장치 화면. 왼쪽 시계는 `2:40`, 오른쪽 출발판은 `3:00`. 아래에는 `10분 부품`이 있다. 아이는 부품을 빈칸에 넣고 `작동`을 누른다.

### 1:30-2:30

10분 부품 1개만 넣으면 버스가 `2:50`까지 가다 멈춘다. 빈칸 하나가 남는다. 아이가 하나 더 넣으면 버스가 3시에 출발하고 시계탑이 움직인다.

### 2:30-4:00

같은 장치를 다른 시간으로 한 번 더 고친다.

```text
4:50 -> 5:00 = 10분 부품 1개
1:30 -> 2:00 = 10분 부품 3개
```

반복이지만 숫자와 장치 반응이 달라야 한다.

### 4:00-5:00

시간 렌즈가 설치된다. 보상 카드가 아니라 실제 토글 버튼이다. 켜면 두 시각 사이의 빈칸이 빛난다.

### 5:00-7:00

짧은 전이 장치. 정류장 전광판이 `5:20 -> 5:40`을 보여준다. 시간 렌즈를 켜고 10분 부품 2개를 넣는다.

### 7:00-8:30

지도 복귀. 시계탑이 실제로 움직이고, 방금 고친 장치를 다시 눌러 작동 재생을 볼 수 있다.

### 8:30-10:00

다음 예고는 추상 미스터리가 아니라 구체 장치다.

```text
버스 번호판: 2, 4, 6, ?
노바: 이번엔 번호 부품이 빠졌어.
```

## 6. 학습 순서

기존 시즌 순서를 다시 정리한다. 초3 회피 학습자는 선수 감각 확인을 시험처럼 보이면 바로 이탈한다. 그래서 입장 감각은 장치 반응으로만 확인한다.

| 우선 | 개념 | 능력 | 수리 장치 |
| --- | --- | --- | --- |
| 0 | 입장 감각 확인 | 노바 관찰 미니 장면 | 시계 읽기, 10씩 세기, 전체/부분 감각을 자연스럽게 확인 |
| 1 | 시각과 시간 차이 | 시간 렌즈 | 시계버스 장치 |
| 2 | 0과 자리값 | 자리 렌즈 | 0이 빠진 번호판 장치 |
| 3 | 수 규칙과 건너세기 | 패턴 렌즈 | 번호 버스 장치 |
| 4 | 덧셈/뺄셈 변화량 | 합치기 장갑, 제거 망치 | 다리 길이와 막힌 골목 장치 |
| 5 | 곱셈의 같은 묶음 | 복제 스탬프 | 좌석, 상자, 가로세로 배열 장치 |
| 6 | 나눗셈의 공평 배분 | 공평 저울 | 접시 배분 장치 |
| 7 | 등분할과 단위분수 | 나누기 칼 | 케이크 분할 장치 |
| 8 | 길이, 넓이, 단위 | 거리 줄자, 복구 타일 | 다리와 바닥 복구 장치 |
| 9 | 도형, 회전, 각 | 모양 도장, 회전 나침반 | 문, 톱니, 거울 장치 |
| 10 | 표와 그래프 | 분석 망원경 | 데이터 항구 증거 장치 |

중요한 변경:

- `사라진 0`은 별도 에피소드로 분리한다.
- `숫자 버스`는 그 다음 규칙 에피소드로 둔다.
- `반쪽 빵집`은 공평 배분과 단위분수를 분리한다.

## 7. 능력 원칙

능력은 도감 보상이 아니라 입력 방식이다.

```text
개념명
→ 능력명
→ 행동 동사
→ 조작 오브젝트
→ 시각 모델
→ 수학 문장
→ 재사용 장소
```

예:

```text
시간 차이
→ 시간 렌즈
→ 두 시각 사이를 본다
→ 10분 부품
→ 2:40~3:00 색 띠
→ 2시 40분 + 20분 = 3시
→ 다른 시간표 장치
```

좋은 능력은 다음 조건을 모두 만족한다.

| 조건 | 기준 |
| --- | --- |
| 드러남 | 켜면 숨은 구조가 보인다 |
| 조작 | 아이가 손으로 부품을 움직인다 |
| 반응 | 맞으면 장치가 움직이고, 틀리면 이유가 보인다 |
| 말하기 | 조작 뒤 짧은 수학 문장으로 정리된다 |
| 재사용 | 다음 장치나 재방문에서 다시 쓰인다 |

## 8. 오개념 처리

오답은 틀림이 아니라 잘못된 생각이 드러난 순간이다.

```text
오답 입력
→ wrongIdea 판정
→ 세계가 이상하게 반응
→ 관찰 힌트
→ 개념 힌트
→ 절차 힌트
→ 같은 장치 재시도
→ 가까운 전이 장치
```

대표 유형:

| 유형 | 예 |
| --- | --- |
| 관찰 누락 | 현재 시각만 보고 출발 시각을 안 봄 |
| 개념 혼동 | 0을 아무것도 없음으로만 봄 |
| 절차 오류 | 10분 칸을 하나 덜 놓음 |
| 표상 오류 | 케이크 조각 수만 같으면 공평하다고 봄 |
| 독해 함정 | 필요한 단서와 장식 정보를 구분하지 못함 |

문구 원칙:

```text
나쁨: 틀렸어요.
좋음: 버스가 너무 일찍 움직였어. 시계에서 3시까지 남은 칸만 다시 봐.
```

오답 횟수, 정답률, 힌트 사용 기록은 저장하지 않는다.

## 9. 저장과 개인정보

저장하지 않는다.

- 이름
- 이메일
- 학년
- 학교
- 보호자 정보
- 원격 학습 기록
- 정답률
- 오답 횟수
- 행동 로그

저장한다.

- 완료한 에피소드 id
- 설치된 능력 id
- 복구된 장치 id
- 발견한 단서 id
- 설정값

모든 저장은 이 브라우저 안 `localStorage`만 사용한다. 브라우저 저장이 사라져도 다시 플레이 가능해야 한다.

## 10. 스택 결정

기존 React/Vite 정적 스택은 유지한다. 문제는 스택이 아니라 화면 문법이었다.

채택:

- React + Vite
- TypeScript 게임 경계
- 상태 있는 inline SVG
- semantic HTML 조작대
- scoped CSS
- localStorage
- GitHub Pages
- Playwright 기반 화면 검증
- registry validation

비채택:

- 자유 이동 RPG 엔진
- Phaser 기본 채택
- Unity
- Three.js 중심 구현
- 서버 저장
- 계정
- service worker 선도입

이유: 핵심은 화려한 엔진이 아니라 상태 있는 수학 오브젝트, 접근성, 모바일, 도서 변환 가능성이다.

## 11. 성공 기준

리부트 PRD는 아래 기준을 통과해야 구현 착수 가능하다.

| 기준 | 통과 조건 |
| --- | --- |
| 첫 30초 | 글 설명 없이 첫 조작을 시도할 수 있다 |
| 첫 2분 | `작동 -> 실패 반응 -> 수정 -> 성공`을 1회 경험한다 |
| 조작 비중 | 읽기 시간보다 장치 조작 시간이 길다 |
| 객관식 의존 | 핵심 미션은 객관식만으로 통과할 수 없다 |
| 오답 반응 | 모든 오답은 장치 반응으로 이유가 보인다 |
| 학습 말하기 | 아이가 “빈칸을 봤다”, “같은 만큼 커졌다”처럼 자기 말로 설명할 수 있다 |
| 전이 | 같은 개념을 다른 장치에 1회 쓴다 |
| 접근성 | 키보드, 터치, 큰 글자, 모션 축소에서 완주 가능하다 |
| 저장 | 진행은 브라우저 안에만 남고, 저장 실패에도 현재 플레이가 멈추지 않는다 |
| 확장 | 시계버스 장치 루프가 통과하기 전에는 에피소드 수를 늘리지 않는다 |

## 12. 작업 순서

1. 기존 시계탑 구현 개선 중단.
2. 라이브 수리 스테이지 graybox 설계.
3. 시계버스 장치만 다시 구현.
4. 첫 2분 루프를 수동 검증.
5. 재미 게이트 통과 후에만 지도, 도감, 단서장 확장.
6. 그 뒤 `사라진 0`, `번호 버스`, `공평 배분`, `단위분수` 순서로 확장.

## 13. 한 줄 판정

시계탑 한 편이 `수학 부품을 끼워 넣어 세계가 작동한다`는 손맛을 증명하기 전에는, 어떤 스토리 확장도 제품 진전으로 보지 않는다.
