# mainPlan - 장기 기획설계 운영 트리

> dartlab의 mainPlan 운영방침을 차용한다(2026-07-10 결정). 제품의 큰 방향 설계는
> 이 트리에서 이니셔티브 단위 폴더로 관리하고, 코드 착수 전 설계의 SSOT가 된다.

## 운영 규칙

1. **이니셔티브 = 폴더 하나.** `mainPlan/<kebab-name>/` 아래 번호 문서(`00-...md`부터)와 `README.md`(인덱스)를 둔다.
2. **문서 구성 관례** (필요한 것만, 순서 고정 아님):
   - `00-product-vision.md` - 무엇을, 누구를 위해, 왜. 기각 근거와 성공/실패 기준.
   - `01-...` 이후 - 아키텍처, 표면 배선, 세부 설계.
   - `killlist-and-non-goals` - 안 하는 것의 명시. 재론 방지.
   - `scope-phasing-guardrails` - phase 분해, 게이트, 테스트 매트릭스, 롤백, 이중 평가.
   - `progress-ledger` - 결정 원장. 세션 간 재개 지점(NEXT)을 항상 최신으로.
3. **비전과 구현이 충돌하면 scope-phasing 문서의 phase 게이트가 우선한다.**
4. **완료·폐기된 이니셔티브는 폴더째 `mainPlan/_done/<name>/`으로 격리한다.**
   삭제하지 않는다(설계 근거·완료 기록·현역 런북은 계속 참조된다). 옮긴 폴더의
   README 상단에 완료/폐기 배너와 한 줄 요약을 남기고, 메모리·외부 참조 경로를 갱신한다.
5. **자기충족성이 합격 기준이다.** 플랜 본문만 보고 재조사 없이 구현 가능해야 한다.
   영향 파일/심볼, 테스트 매핑, 롤백, 평가를 placeholder 없이 채운다.
6. 이 트리는 저장소 규칙(camelCase 파일명)의 예외다. dartlab 관례(kebab-case + 번호 프리픽스)를 따른다.
