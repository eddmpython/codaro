# 11. Beyond-Local Directions - 로컬급을 넘보는 발명급 방향 검토

상태: 탐색 + 부분 실측 v0.1 (2026-07-11). 목표(사용자): 브라우저-서버([10]) 다음으로, 더 혁신적이고 진짜 로컬급이거나 로컬을 초월하는 방향 검토. 발명급.

---

## 0. 출발점: 유일한 약점을 뒤집기
[10]까지 확정: 서버/자동화/파이썬 로직은 브라우저가 로컬급(3.14 인터프리터는 로컬 3.12보다 빠름). **유일한 진짜 약점 = numpy/BLAS 대규모 수치 ~86배 느림**(WASM 단일스레드+128bit SIMD vs 네이티브 멀티코어+AVX). 발명급 방향의 1순위 = 이 약점을 강점으로 뒤집기.

## 1. WebGPU 수치 가속 (로컬 CPU numpy를 이기는 각도)
- 메커니즘: numpy/텐서 연산(matmul/conv/reduction)을 WebGPU compute shader(WGSL)로 GPU에서 실행. Pyodide 파이썬 배열 -> GPU 버퍼 -> 커널 -> 결과 회수. Pyodide는 JS interop로 navigator.gpu 호출 가능.
- 왜 로컬 초월 가능: **로컬 파이썬 numpy는 GPU를 기본으로 안 쓴다(CPU BLAS).** 브라우저는 WebGPU로 GPU에 직접 접근. 큰 행렬곱/딥러닝 추론은 GPU가 CPU를 수십~수백배 이길 수 있어, 브라우저가 로컬 CPU numpy를 이길 여지.
- **실측 상태(정직)**: WebGPU는 2023년부터 정식 API(Chrome 113+). 그러나 **이 headless 환경에선 검증 불가** - GPU 어댑터 없음, 소프트웨어 폴백(swiftshader) 어댑터도 안 잡힘. 즉 "GPU로 numpy 이긴다"는 속도 숫자를 여기서 실증 못 함. 사용자 실제 PC(GPU 있음)에선 동작 예상. 선례: tensorflow.js/ONNX Runtime Web/wonnx가 WebGPU로 GPU 추론 실현.
- 한계: WebGPU 배선 복잡(버퍼/커널/동기화), f32 위주(f64 제한), 작은 연산은 전송 오버헤드로 손해, 브라우저/드라이버 편차. WebNN API(OS 신경망 가속)는 더 이르지만 관찰 대상.

## 2. 실행 환경을 값으로 (로컬이 원리적으로 못 하는 것)
- 메커니즘: **Pyodide 공식 `makeMemorySnapshot` API 존재(실측 확인)** + 검증된 HEAPU8 커스텀 스냅샷([08 계열]). 프레임워크·패키지·데이터까지 로드된 인터프리터 상태를 통째로 스냅샷 값으로 만들어 (a) 즉시 부팅(재-import 스킵), (b) 다른 인스턴스/기기로 전송, (c) fork/시간여행.
- 왜 로컬 초월: **로컬 파이썬은 실행 중 인터프리터를 통째로 스냅샷해 다른 곳에 복원할 수 없다**(Windows fork 없음, CRIU는 리눅스 한정 무거움). 브라우저는 WASM 선형메모리가 ArrayBuffer라 통째 복사/복원이 자명. "실행 환경 = 포터블 값"은 로컬이 못 하는 것.
- 실측 상태: 커스텀 HEAPU8 스냅샷/복원 PASS(리액티브 4배, page-diff 10~14MB, [08 §3.5]). 공식 makeMemorySnapshot 존재 확인. 크로스-인스턴스 전송의 실용 병목 = 크기(전체 ~360MB, page-diff로 셀당 10~14MB).
- 한계: 크기(URL 공유엔 큼, 압축/증분 필요), JS측 핸들(열린 파일/소켓/DOM)은 스냅샷 안 됨(순수 파이썬 상태 한정), emval 경계(별도 연구).

## 3. 추가 방향 전수 검토 (자체 종합)

"브라우저가 로컬을 넘는" 축은 셋으로 갈린다: (A) 속도, (B) 이동성, (C) 안전성. 각 후보를 발명성 x 초월성 x 오늘 실현가능성으로 판정:

| # | 방향 | 축 | 메커니즘 | 판정 |
|---|---|---|---|---|
| 3 | **WebNN** (OS 신경망 가속) | 속도 | 브라우저 표준 NN API가 OS의 NPU/GPU를 씀. ML 추론을 로컬 CPU보다 빠르게 | 발명성 높음. 2026 아직 이른 지원(Chrome 실험). 관찰 대상 |
| 4 | **영속 상태서버(sqlite serialize + IndexedDB/OPFS)** | 이동성/상태 | 서버가 sqlite DB를 바이트로 직렬화(conn.serialize, 3.11+)해 IndexedDB에 영속. 재시작 시 deserialize로 복원 | **실측 PASS(webPyPersist.html)**: 세션1이 3건 쓰고 8192B 영속 -> 페이지 재로드(파이썬 새 부팅) -> 세션2가 복원해 3건+total 12 그대로. **브라우저-서버가 재시작 생존 상주 상태서버가 됨.** OPFS VFS로 대용량 확장 가능 |
| 6 | **WebCodecs** (하드웨어 비디오) | 속도 | 파이썬에서 하드웨어 가속 비디오 인코딩/디코딩. 로컬은 ffmpeg 필요 | 발명성 중. 미디어 자동화 니치. 실현가능 |
| 5 | **상주 데몬**(SW+Periodic Background Sync) | 무인성 | 탭 닫혀도 백그라운드 실행 | **원리적 상한**: 브라우저가 백그라운드 실행을 의도적으로 강력 제한(최소 12h 간격, 설치형 PWA, Chrome 한정). 진짜 set&forget 불가 -> 이건 로컬/Actions 몫 |
| 7 | **즉시 무설치 공유**(스냅샷+URL) | 이동성 | 스냅샷을 링크로 = 상대가 내 실행환경 그대로 실행 | 로컬 초월(로컬은 코드+deps 희망전송뿐). 크기(360MB)가 병목, 증분/압축 필요. #2의 응용 |
| 8 | **샌드박스 안전성** | 안전성 | 브라우저는 신뢰 안 되는 코드를 안전하게 실행. 로컬은 위험 | 학습 제품에 진짜 초월 속성(학생/공유 코드 안전 실행). 속도 아닌 신뢰 축. 이미 자족 티어의 본성 |

## 4. 최종 판정 (검토 완료)

**"브라우저가 로컬을 넘는다"는 세 축 모두에서 실재한다:**
- **속도(수치)**: WebGPU가 numpy 86배 약점을 뒤집는 유일 각도. **실측은 실제 PC(GPU) 필요 - headless 불가**. 최고 payoff, 사용자 PC 검증이 다음.
- **이동성/상태**: 실행환경=값(makeMemorySnapshot 공식+HEAPU8 검증) + OPFS sqlite(오늘 됨)로 상주 상태서버. **로컬이 원리적으로 못 하는 것.**
- **안전성**: 샌드박스는 자족 티어의 본성. 학습/공유 코드 안전 실행 = 로컬 초월 신뢰 속성.

**원리적 상한(정직)**: 진짜 무인 상주 데몬(탭 닫혀도 도는 서버)은 브라우저가 의도적으로 막아 불가 -> 이건 로컬 엔진/GitHub Actions 몫(경계 유지).

**우선순위**:
1. 영속 상태서버(sqlite serialize+IndexedDB) - **실측 PASS(재시작 생존)**. 브라우저-서버가 진짜 상주 상태서버 완성.
2. WebGPU 수치가속 - 최고 초월성, 사용자 실PC(GPU) 실측 필요(headless 불가).
3. 실행환경=값/즉시공유 - makeMemorySnapshot 공식+HEAPU8 검증됨, 크기 병목 해소(증분) 과제.
- 전부 검증된 자산([08]자족·[10]서버·스냅샷)과 결합해 "서버는 로컬급 + 수치는 GPU로 초월 + 환경은 값으로 공유 + 안전은 샌드박스"라는 로컬 초월 그림 완성.
