# 04. Actions Runtime - GitHub Actions를 제3의 파이썬 런타임으로

상태: 상세 설계 v0.1 (2026-07-10)
범위: ActionsEngine 설계, 사용자 소유 리포 모델과 약관 경계, 자동화 승격 레일, 해결/미해결 능력 표, 쿼터·거버넌스.

---

## 1. 위치 정의 (2026-07-10 사용자 결정)

Actions는 cron 편의 기능이 아니라 **WASM 제약을 해결하는 제3의 파이썬 런타임 티어**다. 진짜 리눅스 VM이라 네이티브 휠·subprocess·스레드·headless playwright가 전부 되고, 로컬 없이도("로컬을 떼도") 커리큘럼의 무거운 실행과 자동화 상주가 무료로 산다.

Codaro가 이 티어를 쓸 수 있는 결정적 이유 둘:
1. **리액티브 재실행**: 문서가 의존 그래프라 처음부터 재실행이 의미 보존적이다. 스테이트리스 VM(매 실행 백지)이 Jupyter식 살아있는 커널 모델에는 치명상이지만 Codaro에는 설계 특성과 맞물린다.
2. **percent format**: 노트북이 평범한 .py라서 `uv run python file.py`로 변환 없이 돈다.

## 2. 능력 해결/미해결 표 (정직 상한)

| 능력 | 해결 여부 | 비고 |
|---|---|---|
| 네이티브 휠 (torch·polars·psutil·scikit-learn heavy) | O | pip/uv 전부 |
| subprocess·threading·진짜 소켓 | O | 풀 리눅스 VM |
| headless 브라우저(playwright) | O | 러너에 브라우저 설치 가능 |
| 셀 단위 REPL | X | 콜드스타트+왕복 30~120초. 통실행/부분그래프 재실행 전용 |
| 데스크톱 조작 (xlwings Excel COM·pyautogui) | X | 러너에 Excel·진짜 데스크톱 없음. 영원히 local 전속 |
| 상주 스케줄 | 부분 | cron 최소 5분·혼잡 지연·공개 리포 60일 비활성 시 자동 꺼짐(keepalive 필요) |
| 진짜 내 파일 | X | 데이터는 리포/아티팩트로 왕복해야 함 |

## 3. 소유 모델과 약관 경계 (아키텍처 강제)

- **사용자 소유 리포 모델만 허용한다.** 각 사용자의 노트북·태스크·데이터가 자기 리포에 살고, 그 리포의 Actions가 그 코드를 실행한다. "리포 = 그 사용자의 자동화 프로젝트"라서 약관(리포의 소프트웨어 프로젝트와 무관한 범용 컴퓨트 금지) 안쪽이다.
- **Codaro 중앙 리포로 타인 코드를 실행하는 팜은 금지**(KILL, [05](05-killlist-and-non-goals.md)). 약관 위반이자 격리 문제 재발이다.
- 부수 효과: 컴퓨트도 데이터도 사용자 소유 - local-first 철학의 클라우드 연장.

## 4. ActionsEngine 설계

- ExecutionEngine Protocol 3번째 구현. execute()의 의미를 "부분그래프 배치 재실행"으로 좁힌다(셀 단위 호출이 와도 내부적으로 의존 선행 셀 포함 재실행 계획을 만든다 - kernel/reactive.py의 stale 집합·실행 순서 재사용).
- 흐름: 문서+데이터를 사용자 리포에 커밋 -> workflow_dispatch -> 러너에서 uv 복원(+캐시) -> `python -X utf8 runDocument.py` -> 출력 JSON(_normalizeResult 포맷)을 아티팩트/커밋으로 -> 폴링 회수 -> 셀 결과로 렌더.
- 워크플로 YAML은 Codaro가 생성·커밋해 준다(사용자 수작업 0). setup: checkout + setup-uv + cache.
- 인증: v1 = fine-grained PAT 붙여넣기(브라우저 localStorage, 서버 무경유). OAuth 정식 플로우는 토큰 교환용 초경량 무료 워커가 필요하므로 Phase 5 census 항목으로.
- 실패·지연 UX: "클라우드 실행 중"(경과 표시) -> 타임아웃/실패 시 러너 로그 링크. 조용한 실패 금지(자동화 실패 가시성 원칙과 동형).

## 5. 자동화 승격 레일 (Harvest의 물리적 완성)

- 로컬 자동화 태스크에 "클라우드로 승격" 동작: 태스크 코드 + 스케줄을 workflow YAML(cron)로 변환해 사용자 리포에 푸시. PC가 꺼져 있어도 도는 무료 상주 자동화.
- 리포트 산출물은 리포 커밋 또는 gh-pages 발행, 실패 알림은 기존 아웃바운드 채널(Slack/Discord 웹훅) 재사용.
- 60일 비활성 자동 꺼짐(공개 리포)은 keepalive 커밋 워크플로로 방지하고, 승격 UI에 이 조건을 명시한다.
- 북극성 연결: "검증되는 졸업 자동화"의 종착지. 학습에서 만든 코드가 실제로 무인 상주하는 순간이 졸업이다.

## 6. 쿼터·데이터 거버넌스 (사용자에게 표면화할 것)

- 비공개 리포 무료 = 월 2,000분. 승격 UI가 예상 소비(스케줄 빈도 x 러닝타임)를 보여주고 초과 위험을 경고한다.
- 공개 리포 = 사실상 무제한이나 코드·데이터 공개. 개인정보·업무 데이터가 있는 태스크는 공개 리포 승격을 차단(판정: 데이터 파일 동반 여부 + 사용자 확인).
- 과제방 채점 백엔드 유스케이스는 과제방 보류 결정(2026-07-10)으로 본 PRD 범위에서 제외. 폴더 부활 시 별도 이니셔티브로.

## 7. Phase 5 census (착수 전 검증)

1. workflow_dispatch -> 결과 회수까지 실측 왕복(캐시 유/무), 러너 사양 현행.
2. PAT 최소 권한 셋(contents:write + actions:write 수준) 확정.
3. 공개/비공개 리포별 스케줄 신뢰도(지연 분포)와 60일 규칙 현행 확인.
4. OAuth 교환 워커의 무료 지속 조건(카드 불필요) - 불충족 시 PAT 모델 유지.
